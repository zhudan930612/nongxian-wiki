## 仓库概览

## 页面规范

### YAML Frontmatter

所有知识库页面应包含以下 frontmatter：

```yaml
---
title: 页面标题
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
aliases: [别名1, 别名2]
---
```

可选字段：`source`（来源路径）、`author`（作者）、`status`（active/fading/superseded/archived/draft/absorbed）

### 元数据扩展字段

所有知识库页面可选择性地包含以下扩展字段：

```yaml
confidence: 0.85                        # 总体置信度 [0.0, 1.0]
confidence_factors:                      # 评分明细
  source_count: 8                        # 支持页面的来源数
  last_confirmed: 2026-06-30             # 最近确认日期
  contradiction_count: 0                 # 矛盾数
  cross_references: 5                    # 引用数
supersedes: []                           # 取代的旧页面列表
superseded_by: []                        # 被哪些新页面取代
status: active                           # active / fading / superseded / archived / draft / absorbed（问答页专用）
crystallized_from: query                 # 结晶来源（问答页专有）
session_summary: true                    # 是否完整会话摘要（问答页专有）
```

**字段说明：**

| 字段 | 适用页面 | 说明 |
|------|---------|------|
| `confidence` | 实体/概念/主题/问答 | 0.0-1.0，lint 时批量计算 |
| `confidence_factors` | 同 confidence | 提供评分透明度，lint 时更新 |
| `supersedes` | 所有 | 被当前页面取代的旧页面路径 |
| `superseded_by` | 所有 | 取代当前页面的新页面路径 |
| `status` | 所有 | 可选值：active（活跃）、fading（180天未确认）、superseded（被取代）、archived（归档）、draft（草稿）、absorbed（问答页专用，已吸收到实体/概念页） |
| `crystallized_from` | 问答页 | 结晶来源：query / session / ingest-auto |
| `session_summary` | 问答页 | 是否为完整会话摘要 |

**confidence 计算方式**：

lint 时按以下公式批量计算页面置信度（各因子取值范围 [0.0, 1.0]）：

```
confidence = source_score × 0.30 + timeliness × 0.25 + contradiction_score × 0.20 + reference_score × 0.15 + completeness × 0.10
```

- `source_score`：支持该页面的来源数量与质量
- `timeliness`：信息时效性，`last_confirmed` 越近越高
- `contradiction_score`：与其他来源矛盾程度，矛盾越少越高
- `reference_score`：被其他页面引用的次数与质量
- `completeness`：页面内容完整度

**状态流转：**

```
active ──(180天未确认)──→ fading ──(365天未确认)──→ archived
active ──(被新页面取代)────→ superseded
```

“确认”指用户或 LLM 在 lint/编辑时审阅页面内容并更新 `confidence_factors.last_confirmed`。确认后可逆转 `fading → active`，也可阻止进入 `archived`。

`fading/superseded/archived` 页面默认不参与查询排序，仅在回溯时可见。

## 会话启动流程

每次在 vault 中启动时执行以下检查：

1. 读取 `20-知识库/00-索引/log.md` 最后 5 条记录
2. 检查 `00-收件箱/` 是否有待处理文件 → 有则主动询问
3. 读取 `20-知识库/00-索引/stats.md` 摘要
4. 有 `status: fading` 的页面则提醒用户是否更新

完成后用一两句话告知当前 vault 状态。不要展开。

## 记忆整合层级

知识库的信息按生命周期分为四个层级，从原始观察到固化模式逐级沉淀：

| 层级 | 名称 | 路径 | 说明 |
|------|------|------|------|
| 🟡 **L1 工作记忆** | 原始资料 | `10-原始资料/` | 不可变源文档，LLM 只读不改 |
| 🟠 **L2 情景记忆** | 来源摘要、问答归档 | `04-来源/` `05-问答/` | 结晶的问答和讨论摘要，含待确认事实 |
| 🟢 **L3 语义记忆** | 实体/概念/主题 | `01-实体/` `02-概念/` `03-主题/` | 跨会话的稳定知识，带置信度和取代机制 |
| 🔵 **L4 程序记忆** | Schema/技能 | `CLAUDE.md` `.claude/skills/` | 工作流、模式、约定，定义如何维护前三层 |

**信息晋升路径**：

```
新资料 → L1 工作记忆（收件箱/原始资料）
   ↓ Ingest
L2 情景记忆（来源摘要、问答归档）
   ↓ 事实抽取 + 置信度积累
L3 语义记忆（实体/概念/主题页）
   ↓ 模式提炼
L4 程序记忆（Schema/技能）
```

各层级守则：
- **L1 向 L2 晋升**：每次 Ingest 自动完成，来源摘要页引用实体和概念
- **L2 向 L3 晋升**：结晶（问答/长讨论）中的事实原子经 lint 确认后吸收到实体/概念页
- **L3 向 L4 晋升**：当发现可复用的工作流或模式时，更新 CLAUDE.md 或 skills

## 三大核心工作流

### 1️⃣ 摄取 (Ingest)

当用户在 `00-收件箱/` 放入新资料并指示 LLM 处理时（如"读取"、"分析"、"提炼"、"整理"、"总结"），自动调用 `llm-wiki-ingest` skill 执行完整摄取流程。

> 注：Ingest 阶段的链接可指向不存在的页面（前向引用），灰色链接由 lint 阶段集中处理。详见 [[20-知识库/06-元/06-约定/forward-reference-convention|前向引用约定]]。

> 如果检测到重复、资料无正文内容，或分类边界不明确需要用户确认，则暂停或跳过后续步骤。详细执行规范见 `llm-wiki-ingest` skill。

#### 关于 `00-收件箱/`

`00-收件箱/` 是 **唯一入口**。所有新增资料——无论是 Obsidian Web Clipper 剪藏、手动复制文件、PDF、图片、语音转文字、会议速记、灵感笔记——**全部先丢到这里**。

你不需要自己分类。只要说"读取"、"分析"、"提炼"、"整理"、"总结"等，LLM 会：

- 判断资料类型
- 迁移到 `10-原始资料/` 下对应分类目录（政策法规/学术论文/行业报告/技术规范/产品与公司/媒体报道/会议沟通）
- 在知识库中创建摘要、实体、概念、主题页面
- 更新索引、日志、统计
- 抽取知识图谱关系（在来源正文中识别实体间关系，追加到 `20-知识库/00-索引/knowledge-graph.md`）
- 最后清理收件箱

这保证了整个工作流只有一个入口，你几乎不用做维护。

**收件箱忽略规则**：以下文件/目录不视为待处理资料，ingest 和启动检查自动跳过：

- `README.md`、`.gitkeep` 等说明/占位文件
- 以 `.` 开头的隐藏文件/目录
- `PDF解析/` 等 ingest 临时输出目录
- 空文件或仅有 frontmatter 的占位文件

### 2️⃣ 查询 (Query)

当用户向知识库提问时（如"查询"、"搜索"、"告诉我关于"），自动调用 `llm-wiki-query` skill 执行检索、综合回答与结晶归档。详细执行规范见 `.claude/skills/llm-wiki-query/SKILL.md`。

#### 结晶触发条件

结晶不限于问答场景。满足以下任一条件时自动创建结晶归档：

- **综合性分析**：回答涉及 3+ 个不同来源的交叉分析
- **新颖结论**：包含知识库现有页面未覆盖的观点
- **长讨论自动化**：一次对话涉及 5 轮以上且产生可复用结论时，主动创建结晶归档
- **关键决策**：讨论中达成涉及项目方向的共识或设计决策
- **高频率**：同一类型问题被多次询问

### 3️⃣ 检查与自愈 (Lint + Self-heal)

当用户要求对知识库进行健康检查、审计或维护 review 时（如"检查"、"lint"、"审计"、"健康检查"、"整理知识库"），自动调用 `llm-wiki-lint` skill 执行检查与自愈。

检查结果以报告形式输出，并更新 `20-知识库/00-索引/stats.md` 中的"最后检查"日期。详细执行规范见 `llm-wiki-lint` skill。

建议检查频率：
- 每摄取 5–10 个资料后执行一次
- 导出或重大回顾前执行一次
- 用户主动要求时执行

#### Lint 扩展步骤

完整 lint 扩展步骤（置信度批量评分、取代检测、结晶后处理、知识图谱验证）定义于 `.claude/skills/llm-wiki-lint/SKILL.md`。

## 迭代原则

- 接受"不完美"——首次创建的页面不必详尽，随知识积累逐步完善
- 本 Schema 文件会随实际使用持续进化
- 当发现工作流不合理时，用户可以要求修改本文件
