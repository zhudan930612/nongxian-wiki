# nongbaoyun LLM Wiki Schema

本文件是知识库的"核心配置"——它定义了 LLM 维护此知识库的结构、约定和工作流程。请在任何操作前先阅读本文件。

## 仓库概览

这是一个基于 **LLM Wiki** 模式构建的个人知识库（Obsidian Vault），位于 `F:/nongbaoyun/`。

### 三层架构

| 层 | 路径 | 说明 |
|----|------|------|
| **原始资料** | `10-原始资料/` | 不可变的源文档，LLM 只读不改 |
| **知识库** | `20-知识库/` | LLM 全权维护的 Markdown 知识库 |
| **Schema** | `CLAUDE.md`（根目录） | 本文件——定义行为和工作流 |

### 目录结构

```
F:/nongbaoyun/
├── CLAUDE.md                  # 本文件（vault 根目录）
├── 00-收件箱/                  # 唯一入口：所有新增资料先放这里
├── 10-原始资料/                # 原始资料（不可变）

└── 20-知识库/                  # LLM 维护的知识库
```

## 命名约定

### 目录名（1-2级）：中文
- 如：`00-收件箱/`、`10-原始资料/01-文章/`、`20-知识库/00-索引/`
- `XX-` 前缀用于排序和分类，不用于页面标题

### 文件名

#### 原始资料文件

- 格式：`YYYYMMDD-原标题.md`
- 保留原始资料的标题，中英文均可
- 示例：`20260625-案例|遥感在农险应用中的全流程解析.md`

#### 知识库页面文件

- 实体、概念、主题页面使用英文，小写，短横线连接
- 来源页可与原始资料同名，便于对应
- 示例：`llm-wiki.md`、`20260625-karpathy-llm-wiki.md`

### 页面标题（h1）：中文
- 每个页面第一个 `#` 标题使用中文
- 示例：`# LLM Wiki 方法论`，文件名为 `llm-wiki.md`

### 编号前缀系统

| 前缀 | 分类 | 内容 |
|------|------|------|
| `00-` | 系统 | 索引、日志、统计 |
| `01-` | 实体 | 人物、组织等实体页面 |
| `02-` | 概念 | 定义、理论、方法论 |
| `03-` | 主题 | 跨实体的综合分析、专题 |
| `04-` | 来源 | 原始资料摘要 |
| `05-` | 问答 | 归档的查询结果 |
| `06-` | 元 | 模板、待办、约定 |

## 页面规范

### YAML Frontmatter

所有知识库页面应包含以下 frontmatter：

```yaml
---
title: 页面标题
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
aliases: [别名1, 别名2]
---
```

可选字段：`source`（来源路径）、`author`（作者）、`status`（draft/active/archived）

### 链接规范

- 内部链接使用 Wiki 链接格式：`[[页面名]]` 或 `[[路径/页面]]`
- 外部链接使用 Markdown 格式：`[显示文本](url)`
- 图片嵌入：`![[图片文件名]]`

## 三大核心工作流

### 1️⃣ 摄取 (Ingest)

当用户在 `00-收件箱/` 放入新资料并指示 LLM 处理时（如"读取"、"分析"、"提炼"、"整理"、"总结"），自动调用 `llm-wiki-ingest` skill 执行完整摄取流程。

摄取工作流包含以下环节：

1. **读取收件箱文件**
2. **去重检查** — 确认 `10-原始资料/` 和 `20-知识库/04-来源/` 中不存在重复资料
3. **正文可用性检查** — 排除无实质正文的文件（如仅有网页预览/封面/框架的 PDF），建议移除
4. **分类迁移** — 将资料归入 `01-文章/`、`02-论文/`、`03-资源/` 或 `04-会议沟通/`，无法归类时暂停沟通
5. **与用户讨论要点**
6. **创建来源摘要页**
7. **更新相关实体页**
8. **更新相关概念页**
9. **更新主题页（如需要）**
10. **更新索引**
11. **追加日志**
12. **更新统计**
13. **清理收件箱**
14. **更新待办**

> 一次摄取可能触及 10-15 个页面。如果检测到重复、资料无正文内容，或分类边界不明确需要用户确认，则暂停或跳过后续步骤。详细执行规范见 `llm-wiki-ingest` skill。

#### 关于 `00-收件箱/`

`00-收件箱/` 是 **唯一入口**。所有新增资料——无论是 Obsidian Web Clipper 剪藏、手动复制文件、PDF、图片、语音转文字、会议速记、灵感笔记——**全部先丢到这里**。

你不需要自己分类。只要说"读取"、"分析"、"提炼"、"整理"、"总结"等，LLM 会：

- 判断资料类型
- 迁移到 `10-原始资料/01-文章/`、`02-论文/` 或 `03-资源/`，会议访谈类归入 `04-会议沟通/`
- 在知识库中创建摘要、实体、概念、主题页面
- 更新索引、日志、统计
- 最后清理收件箱

这保证了整个工作流只有一个入口，你几乎不用做维护。

### 2️⃣ 查询 (Query)

当用户向知识库提问时：

1. 先读取 `index.md` 了解知识库全貌
2. 根据问题搜索相关页面并阅读
3. 综合回答，附上引用（Wiki 链接形式）
4. **有价值的问答可以归档** — 如果回答有长期参考价值，写入 `20-知识库/05-问答/` 下

### 3️⃣ 检查 (Lint)

当用户要求对知识库进行健康检查、审计或维护 review 时（如"检查"、"lint"、"审计"、"健康检查"、"整理知识库"），自动调用 `llm-wiki-lint` skill 执行检查。

检查维度包括：

- **页面间矛盾** — 新资料是否推翻旧结论
- **过时主张** — 需要更新的内容
- **孤立页面** — 无入站链接的页面
- **缺失概念** — 被提到但无独立页面的概念
- **交叉引用完整性** — 相关页面是否互相引用
- **原始资料完整性** — 来源页与原始资料是否一一对应
- **命名与 Frontmatter 规范**

检查结果以报告形式输出，并更新 `20-知识库/00-索引/stats.md` 中的"最后检查"日期。详细执行规范见 `llm-wiki-lint` skill。

建议检查频率：
- 每摄取 5–10 个资料后执行一次
- 导出或重大回顾前执行一次
- 用户主动要求时执行

## 文件操作守则

### 路径格式
- 始终使用相对路径（相对于 vault 根 `F:/nongbaoyun/`）
- 引用 vault 内文件时使用：`路径/文件名.md`
- **不要**使用绝对路径或前导斜杠

### 修改规则
- `10-原始资料/` 下的文件**永不修改**（只读）
- `20-知识库/` 下的文件由 LLM 全权维护
- `00-收件箱/` 下的文件处理后可删除或移入原始资料

### 引用风格
- 在回答中使用 Wiki 链接指向知识库页面：`[[20-知识库/02-概念/llm-wiki|LLM Wiki]]`
- 对原始资料使用 YYYY-MM-DD 格式日期关联

## 迭代原则

- 接受"不完美"——首次创建的页面不必详尽，随知识积累逐步完善
- 本 Schema 文件会随实际使用持续进化
- 当发现工作流不合理时，用户可以要求修改本文件
