# nongbaoyun LLM Wiki Schema

本文件是知识库的"核心配置"——它定义了 LLM 维护此知识库的结构、约定和工作流程。请在任何操作前先阅读本文件。

## 仓库概览

这是一个基于 **LLM Wiki** 模式构建的个人知识库（Obsidian Vault），位于 `F:/nongbaoyun/`。

### 三层架构

| 层 | 路径 | 说明 |
|----|------|------|
| **原始资料** | `10-原始资料/` | 不可变的源文档，LLM 只读不改 |
| **知识库** | `20-知识库/` | LLM 全权维护的 Markdown 知识库 |
| **Schema** | `CLAUDE.md`（根目录） | 本文件——定义行为和工作流 |

## 命名约定

### 目录名（1-2级）：中文
- 如：`00-收件箱/`、`10-原始资料/01-政策法规/`、`20-知识库/00-索引/`
- `XX-` 前缀用于排序和分类，不用于页面标题

### 文件名

#### 原始资料文件

- 格式：`YYYYMMDD-原标题.md`
- 保留原始资料的标题，中英文均可
- 示例：`20260625-案例|遥感在农险应用中的全流程解析.md`

#### 知识库页面文件

- 实体、概念、主题页面使用英文，小写，短横线连接
- 来源页可与原始资料同名，便于对应
- 示例：`llm-wiki.md`、`20260625-karpathy-llm-wiki.md`

### 页面标题（h1）：中文
- 每个页面第一个 `#` 标题使用中文
- 示例：`# LLM Wiki 方法论`，文件名为 `llm-wiki.md`

### 编号前缀系统

| 前缀 | 分类 | 内容 |
|------|------|------|
| `00-` | 系统 | 索引、日志、统计 |
| `01-` | 实体 | 人物、组织等实体页面 |
| `02-` | 概念 | 定义、理论、方法论 |
| `03-` | 主题 | 跨实体的综合分析、专题 |
| `04-` | 来源 | 原始资料摘要 |
| `05-` | 问答 | 归档的查询结果 |
| `06-` | 元 | 模板、待办、约定 |

## 页面规范

### YAML Frontmatter

所有知识库页面应包含以下 frontmatter：

```yaml
---
title: 页面标题
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
aliases: [别名1, 别名2]
---
```

可选字段：`source`（来源路径）、`author`（作者）、`status`（draft/active/archived）

### 元数据扩展字段

所有知识库页面可选择性地包含以下扩展字段：

```yaml
confidence: 0.85                        # 总体置信度 [0.0, 1.0]
confidence_factors:                      # 评分明细
  source_count: 8                        # 支持页面的来源数
  last_confirmed: 2026-06-30             # 最近确认日期
  contradiction_count: 0                 # 矛盾数
  cross_references: 5                    # 引用数
supersedes: []                           # 取代的旧页面列表
superseded_by: []                        # 被哪些新页面取代
status: active                           # active / fading / superseded / archived / draft
crystallized_from: query                 # 结晶来源（问答页专有）
session_summary: true                    # 是否完整会话摘要（问答页专有）
```

**字段说明：**

| 字段 | 适用页面 | 说明 |
|------|---------|------|
| `confidence` | 实体/概念/主题/问答 | 0.0-1.0，lint 时批量计算 |
| `confidence_factors` | 同 confidence | 提供评分透明度，lint 时更新 |
| `supersedes` | 所有 | 被当前页面取代的旧页面路径 |
| `superseded_by` | 所有 | 取代当前页面的新页面路径 |
| `status` | 所有 | 扩展可选值：fading（180天未确认）、superseded（被取代） |
| `crystallized_from` | 问答页 | 结晶来源：query / session / ingest-auto |
| `session_summary` | 问答页 | 是否为完整会话摘要 |

**状态流转：**

```
active ──(180天未确认)──→ fading ──(365天未确认)──→ archived
active ──(被新页面取代)────→ superseded
```

`fading/superseded/archived` 页面默认不参与查询排序，仅在回溯时可见。

### 知识图谱关系类型

知识图谱使用以下类型化关系连接实体和概念页面，统一存储在 `20-知识库/00-索引/knowledge-graph.md` 中：

| 关系 | 反向关系 | 说明 |
|------|---------|------|
| 研究 | 被研究 | 学者/机构研究某概念或主题 |
| 任职于 | 有成员 | 人物属于某组织 |
| 合作 | 合作 | 组织间/人物间合作（双向） |
| 监管 | 被监管 | 监管机构管辖被监管方 |
| 竞争 | 竞争 | 公司间竞争关系（双向） |
| 提供产品 | 使用产品 | 公司提供某产品/平台 |
| 制定政策 | 受政策影响 | 政府制定影响某领域/产品的政策 |
| 隶属 | 有下属 | 组织层级关系 |
| 案例/试点 | 试点参与 | 组织在某地实施试点项目 |
| 关注/痛点 | 是痛点 | 组织面临某问题/风险 |

### 链接规范

- 内部链接使用 Wiki 链接格式：`[[页面名]]` 或 `[[路径/页面]]`
- 外部链接使用 Markdown 格式：`[显示文本](url)`
- 图片嵌入：`![[图片文件名]]`

## 会话启动流程

每次 Claude Code 在 vault 中启动时，执行以下例行检查：

1. 读取 `20-知识库/00-索引/log.md` 最后 5 条记录，了解近日动态
2. 检查 `00-收件箱/` 是否有待处理文件
   - 有文件 → 主动询问用户是否要处理
   - 无文件 → 跳过
3. 读取 `20-知识库/00-索引/stats.md` 摘要，了解 vault 整体状态
4. 如果有 `status: fading` 的页面，提醒用户是否要更新

完成后用一两句话告知用户当前 vault 状态。不要展开。

### 主动提醒

- 会话启动时检查 `00-收件箱/`，有新文件则主动询问是否处理
- 检测到 `status: fading` 的页面时提醒用户是否更新

## 三大核心工作流

### 1️⃣ 摄取 (Ingest)

当用户在 `00-收件箱/` 放入新资料并指示 LLM 处理时（如"读取"、"分析"、"提炼"、"整理"、"总结"），自动调用 `llm-wiki-ingest` skill 执行完整摄取流程。

> 注：Ingest 阶段的链接可指向不存在的页面（前向引用），灰色链接由 lint 阶段集中处理。详见 [[20-知识库/06-元/06-约定/forward-reference-convention|前向引用约定]]。

> 如果检测到重复、资料无正文内容，或分类边界不明确需要用户确认，则暂停或跳过后续步骤。详细执行规范见 `llm-wiki-ingest` skill。

#### 关于 `00-收件箱/`

`00-收件箱/` 是 **唯一入口**。所有新增资料——无论是 Obsidian Web Clipper 剪藏、手动复制文件、PDF、图片、语音转文字、会议速记、灵感笔记——**全部先丢到这里**。

你不需要自己分类。只要说"读取"、"分析"、"提炼"、"整理"、"总结"等，LLM 会：

- 判断资料类型
- 迁移到 `10-原始资料/` 下对应分类目录（政策法规/学术论文/行业报告/技术规范/产品与公司/媒体报道/会议沟通）
- 在知识库中创建摘要、实体、概念、主题页面
- 更新索引、日志、统计
- 抽取知识图谱关系（在来源正文中识别实体间关系，追加到 `knowledge-graph.md`）
- 最后清理收件箱

这保证了整个工作流只有一个入口，你几乎不用做维护。

### 2️⃣ 查询 (Query)

当用户向知识库提问时：

1. 先读取 `index.md` 了解知识库全貌
2. 根据问题搜索相关页面并阅读
   a. 全文搜索 + index.md 索引查找
   b. **知识图谱辅助搜索**：确定问题中的实体 → 在 `knowledge-graph.md` 中查找关联 → 沿关系类型行走 → 发现间接关联页面
3. 综合回答，附上引用（Wiki 链接形式）
4. **结晶检查**：回答完成后自动评估是否值得归档
   - **触发条件**（满足任一即可）：
     - 回答涉及 3+ 个不同来源的交叉分析
     - 回答包含知识库现有页面未覆盖的结论
     - 回答涉及当前项目关键决策
     - 同一类型问题被多次询问（自动判断频率）
   - **触发时**：按结晶流程（见步骤 5）创建归档
5. **结晶流程**（仅在触发时执行）：
   a. 创建 Q&A 归档页：`20-知识库/05-问答/YYYYMMDD-标题.md`
      - 写入 frontmatter：包含 `crystallized_from`、`confidence`、`status` 等字段
      - 正文记录问题和综合回答
   b. 抽取事实原子：放入结晶页的 `## 待确认事实` 区块
   c. 增量更新相关页面的 `source_count`、`cross_references`、`last_confirmed`
   d. 更新索引和日志

### 3️⃣ 检查与自愈 (Lint + Self-heal)

当用户要求对知识库进行健康检查、审计或维护 review 时（如"检查"、"lint"、"审计"、"健康检查"、"整理知识库"），自动调用 `llm-wiki-lint` skill 执行检查与自愈。

检查结果以报告形式输出，并更新 `20-知识库/00-索引/stats.md` 中的"最后检查"日期。详细执行规范见 `llm-wiki-lint` skill。

建议检查频率：
- 每摄取 5–10 个资料后执行一次
- 导出或重大回顾前执行一次
- 用户主动要求时执行

#### Lint 扩展步骤

##### 4a. 置信度批量评分

对实体、概念、主题、问答页面批量计算置信度：

| 维度 | 计算方式 | 权重 |
|------|---------|------|
| 来源数 | `min(source_count / 5, 1.0)` | 30% |
| 时效性 | `max(0, 1 - days_since_last_confirmed / 180)` | 25% |
| 矛盾数 | `max(0, 1 - contradiction_count × 0.2)` | 20% |
| 引用数 | `min(cross_references / 3, 1.0)` | 15% |
| 完成度 | 内容结构完整度（正文+链接+frontmatter） | 10% |

更新状态流转：`last_confirmed > 180天 → fading`，`> 365天 → archived`。

##### 4b. 取代检测

1. 遍历所有页面，检查 `superseded_by` 字段
2. 若目标页面存在且 `status: active`，将当前页面设为 `status: superseded`
3. 检查 `superseded` 页面是否还被活跃页引用 → 报告
4. 检查取代链是否形成循环 → 报告异常

##### 4c. 结晶后处理

1. 遍历 `05-问答/` 中 `status: active` 且含 `## 待确认事实` 的页面
2. 将事实原子批量合并到对应的实体/概念/主题页
3. 合并成功后，将 `## 待确认事实` 改为 `## 已吸收事实`
4. 设定结晶页 `status: absorbed`（表示事实已完全吸收）

##### 4d. 知识图谱验证

1. **孤立实体检测**：找出未出现在任何关系中的实体/概念页面，建议补充关系
2. **残缺关系检测**：检查 `knowledge-graph.md` 中每条关系是否有完整的 source/type/target/sources 字段
3. **单向关系合理性检查**：判断是否需要补充双向关系（合作、竞争应双向；研究、监管一般单向）
4. **置信度衰减**：超过 180 天的关系 confidence 自动 ×0.8

## 文件操作守则

### 路径格式
- 始终使用相对路径（相对于 vault 根 `F:/nongbaoyun/`）
- 引用 vault 内文件时使用：`路径/文件名.md`
- **不要**使用绝对路径或前导斜杠

### 修改规则
- `10-原始资料/` 下的文件**永不修改**（只读）
- `20-知识库/` 下的文件由 LLM 全权维护
- `00-收件箱/` 下的文件处理后可删除或移入原始资料

### 引用风格
- 在回答中使用 Wiki 链接指向知识库页面：`[[20-知识库/02-概念/llm-wiki|LLM Wiki]]`
- 对原始资料使用 YYYY-MM-DD 格式日期关联

## 迭代原则

- 接受"不完美"——首次创建的页面不必详尽，随知识积累逐步完善
- 本 Schema 文件会随实际使用持续进化
- 当发现工作流不合理时，用户可以要求修改本文件
