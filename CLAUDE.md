# nongbaoyun LLM Wiki Schema

本文件是知识库的"核心配置"——它定义了 LLM 维护此知识库的结构、约定和工作流程。请在任何操作前先阅读本文件。

## 仓库概览

这是一个基于 **LLM Wiki** 模式构建的个人知识库（Obsidian Vault），位于 `F:/nongbaoyun/`。

### 三层架构

| 层 | 路径 | 说明 |
|----|------|------|
| **原始资料** | `10-原始资料/` | 不可变的源文档，LLM 只读不改 |
| **知识库** | `20-知识库/` | LLM 全权维护的 Markdown 知识库 |
| **Schema** | `CLAUDE.md`（根目录） | 本文件——定义行为和工作流 |

### 目录结构

```
F:/nongbaoyun/
├── CLAUDE.md                  # 本文件（vault 根目录）
├── 00-收件箱/                  # 唯一入口：所有新增资料先放这里
├── 10-原始资料/                # 原始资料（不可变）
│   ├── 01-文章/                # 网页文章、博客
│   ├── 02-论文/                # 学术论文
│   └── 03-资源/                # 图片、PDF 等
└── 20-知识库/                  # LLM 维护的知识库
    ├── 00-索引/                # 索引、日志、统计
    │   ├── index.md
    │   ├── log.md
    │   └── stats.md
    ├── 01-实体/                # 人物/组织实体
    ├── 02-概念/                # 核心概念
    ├── 03-主题/                # 综合主题分析
    ├── 04-来源/                # 原始资料摘要
    ├── 05-问答/                # 归档的高质量问答
```

## 命名约定

### 目录名（1-2级）：中文
- 如：`00-收件箱/`、`10-原始资料/01-文章/`、`20-知识库/00-索引/`
- `XX-` 前缀用于排序和分类，不用于页面标题

### 文件名

#### 原始资料文件

- 格式：`YYYYMMDD-原标题.md`
- 保留原始资料的标题，中英文均可
- 示例：`20260625-案例|遥感在农险应用中的全流程解析.md`

#### 知识库页面文件

- 实体、概念、主题页面使用英文，小写，短横线连接
- 来源页可与原始资料同名，便于对应
- 示例：`llm-wiki.md`、`20260625-karpathy-llm-wiki.md`

### 页面标题（h1）：中文
- 每个页面第一个 `#` 标题使用中文
- 示例：`# LLM Wiki 方法论`，文件名为 `llm-wiki.md`

### 编号前缀系统

| 前缀 | 分类 | 内容 |
|------|------|------|
| `00-` | 系统 | 索引、日志、统计 |
| `01-` | 实体 | 人物、组织等实体页面 |
| `02-` | 概念 | 定义、理论、方法论 |
| `03-` | 主题 | 跨实体的综合分析、专题 |
| `04-` | 来源 | 原始资料摘要 |
| `05-` | 问答 | 归档的查询结果 |
| `06-` | 元 | 模板、待办、约定 |

## 页面规范

### YAML Frontmatter

所有知识库页面应包含以下 frontmatter：

```yaml
---
title: 页面标题
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
aliases: [别名1, 别名2]
---
```

可选字段：`source`（来源路径）、`author`（作者）、`status`（draft/active/archived）

### 链接规范

- 内部链接使用 Wiki 链接格式：`[[页面名]]` 或 `[[路径/页面]]`
- 外部链接使用 Markdown 格式：`[显示文本](url)`
- 图片嵌入：`![[图片文件名]]`

## 三大核心工作流

### 1️⃣ 摄取 (Ingest)

当用户在 `00-收件箱/` 放入新资料并指示 LLM 处理时（如"读取"、"分析"、"提炼"、"整理"、"总结"）：

1. **读取收件箱文件** — 完整阅读 `00-收件箱/` 中的源文件
2. **去重检查** — 在处理前确认该资料是否已存在：
   - 检查 `10-原始资料/` 中是否已有**同名文件**
   - 检查 `20-知识库/04-来源/` 中是否已有**内容高度相似的来源页**（对比标题、摘要、关键段落）
   - 检查文件 frontmatter 中的 `source` URL 是否已存在
   - 如果判定为重复，提示用户并跳过摄取，不创建新页面
3. **分类迁移** — LLM 根据内容判断类型，将原始资料迁移到 `10-原始资料/` 下的对应目录：
   - 网页文章/博客 → `10-原始资料/01-文章/`
   - 学术论文 → `10-原始资料/02-论文/`
   - 图片/PDF/数据 → `10-原始资料/03-资源/`
   - 用户自己的笔记/灵感 → 直接在知识库中创建或补充页面
   重命名为规范格式：`YYYYMMDD-原标题.md`
4. **与用户讨论要点** — 确认核心观点和关注方向
5. **创建来源摘要页** — 在 `20-知识库/04-来源/` 下创建与原始资料同名的摘要页
6. **更新相关实体页** — 在 `20-知识库/01-实体/` 更新或创建人物/组织页面
7. **更新相关概念页** — 在 `20-知识库/02-概念/` 更新或创建概念页面
8. **更新主题页（如需要）** — 在 `20-知识库/03-主题/` 更新综合分析
9. **更新索引** — 修改 `20-知识库/00-索引/index.md` 添加条目
10. **追加日志** — 在 `20-知识库/00-索引/log.md` 添加记录
11. **更新统计** — 在 `20-知识库/00-索引/stats.md` 更新计数
12. **清理收件箱** — 从 `00-收件箱/` 移除已处理的文件
13. **更新待办** — 在 `20-知识库/06-元/06-todo.md` 清理相关项

> 一次摄取可能触及 10-15 个页面。如果检测到重复，则跳过后续步骤。

#### 关于 `00-收件箱/`

`00-收件箱/` 是 **唯一入口**。所有新增资料——无论是 Obsidian Web Clipper 剪藏、手动复制文件、PDF、图片、语音转文字、会议速记、灵感笔记——**全部先丢到这里**。

你不需要自己分类。只要说"读取"、"分析"、"提炼"、"整理"、"总结"等，LLM 会：

- 判断资料类型
- 迁移到 `10-原始资料/01-文章/`、`02-论文/` 或 `03-资源/`
- 在知识库中创建摘要、实体、概念、主题页面
- 更新索引、日志、统计
- 最后清理收件箱

这保证了整个工作流只有一个入口，你几乎不用做维护。

### 2️⃣ 查询 (Query)

当用户向知识库提问时：

1. 先读取 `index.md` 了解知识库全貌
2. 根据问题搜索相关页面并阅读
3. 综合回答，附上引用（Wiki 链接形式）
4. **有价值的问答可以归档** — 如果回答有长期参考价值，写入 `20-知识库/05-问答/` 下

### 3️⃣ 检查 (Lint)

定期执行的知识库健康检查：

- 检查页面间矛盾（新资料是否推翻旧结论）
- 检查过时主张（需要更新的内容）
- 检查孤立页面（无入站链接的页面）
- 检查缺失概念（被提到但无独立页面的概念）
- 检查交叉引用完整性
- 建议新的探索方向

## 文件操作守则

### 路径格式
- 始终使用相对路径（相对于 vault 根 `F:/nongbaoyun/`）
- 引用 vault 内文件时使用：`路径/文件名.md`
- **不要**使用绝对路径或前导斜杠

### 修改规则
- `10-原始资料/` 下的文件**永不修改**（只读）
- `20-知识库/` 下的文件由 LLM 全权维护
- `00-收件箱/` 下的文件处理后可删除或移入原始资料

### 引用风格
- 在回答中使用 Wiki 链接指向知识库页面：`[[20-知识库/02-概念/llm-wiki|LLM Wiki]]`
- 对原始资料使用 YYYY-MM-DD 格式日期关联

## 迭代原则

- 接受"不完美"——首次创建的页面不必详尽，随知识积累逐步完善
- 本 Schema 文件会随实际使用持续进化
- 当发现工作流不合理时，用户可以要求修改本文件
